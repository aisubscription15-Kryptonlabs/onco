Local STT for Telugu — what could replace Sarvam in MedAssist

Comparison of locally-runnable speech-to-text models for Telugu (and code-mixed Telugu/English) — accuracy, hardware needs, diarisation support, and what it would take to swap into MedAssist's transcription pipeline.

Research-only. No code changes have been made.

TL;DR. Three options realistically replace Sarvam for Telugu today: (1) AI4Bharat IndicConformer / IndicWhisper for the highest Telugu accuracy and best code-mixed handling, (2) NVIDIA NeMo Parakeet-TDT-Indic for the fastest GPU inference, and (3) faster-whisper Large-v3 as a no-fine-tuning baseline. Diarisation is not built into any of these — it's a separate add-on (typically pyannote.audio). Plan for a Linux box with at least one GPU; CPU-only is feasible but slow on long visits.

What we need from the replacement

The MedAssist pipeline expects three things from one transcription call:

  • Telugu (and Hindi/English code-mixed) ASR — Indian doctor-patient consultations switch languages mid-sentence. Pure-English models drop accuracy below useful.
  • Translation to English — Claude does extraction in English. The transcript fed to Claude is translated.
  • Speaker diarisation — labels like SPEAKER_1 / SPEAKER_2. Claude uses these to figure out which voice is the doctor.

No single open-source model does all three. A local replacement is a small pipeline: ASR → (optional MT) → diarisation, then merged into the same JSON shape the Sarvam wrapper produces today ({ transcript, language_code, turns: [...] }).

Recommended pipeline (highest accuracy)

1. ASR — IndicConformer (or IndicWhisper)

AI4Bharat's indic-conformer-600m-multilingual is the strongest Telugu ASR available open-source today. It outperforms Whisper-large-v3 by a wide margin on Telugu and handles code-mixing because the training data is code-mixed Indian speech, not translated English. Output is in Telugu script (Devanagari + ITRANS for code-switch points).

2. MT — IndicTrans2 (Telugu → English)

AI4Bharat's indictrans2-indic-en-1B translates the Telugu transcript to English so Claude can extract fields. Quality is on par with commercial services for medical content (BLEU ~38 on FLORES-Telugu). For code-mixed input, IndicTrans2 already passes English tokens through unchanged, which is exactly what we want for drug names and BD/TDS shorthand.

3. Diarisation — pyannote.audio 3.x

pyannote/speaker-diarization-3.1 is the de-facto open-source diarisation system. It produces speaker-labelled time spans (start, end, SPEAKER_00, …) which get merged with the ASR transcript to assemble the same turns: [...] array Sarvam returns today.

Pre-set num_speakers=2 when known (most OPD visits) — gives a noticeable accuracy bump.

Hardware sizing

Minimum (single doctor, casual use)

  • RTX 3060 12 GB (~₹30k used) or RTX 4060 16 GB
  • 32 GB system RAM, NVMe SSD
  • Linux (Ubuntu 22.04+) — Docker-compose with NVIDIA Container Toolkit
  • One 5-min visit transcribed end-to-end in ≈45 s
Cost ≈ ₹70k one-time, ₹0 per minute

Recommended (clinic with 3 doctors in parallel)

  • RTX 4090 24 GB or A6000 48 GB
  • 64 GB system RAM, dual NVMe
  • vLLM-style batching for ASR; pyannote in a separate container
  • 3 concurrent 5-min visits in ≈30 s (with Parakeet-TDT)
Cost ≈ ₹2.5–3.5L one-time

What it would cost to swap into MedAssist

The wrapper at lib/stt/sarvam.ts already returns a clean SarvamResult with { transcript, language_code, turns }. A local pipeline replaces only the implementation, not the contract. New file lib/stt/local.ts calls a self-hosted FastAPI/Triton service and returns the same shape; app/api/transcribe/route.ts picks one based on an env flag.

Effort estimate, in person-days:

  • Day 1–2: Containerise the three models (ASR, MT, diarisation) behind one FastAPI service. Use NeMo for IndicConformer, HF for IndicTrans2, pyannote-audio for diarisation. One POST /transcribe endpoint that takes an audio URL.
  • Day 3: Add merge logic — align word-level timestamps from ASR with pyannote spans to produce turns[] in the exact Sarvam shape.
  • Day 4: Add STT_PROVIDER=local|sarvam env switch in lib/env.ts; new lib/stt/local.ts that calls the FastAPI service. Update app/api/transcribe/route.ts to pick the provider.
  • Day 5: Side-by-side benchmark on 10 real consultation recordings — pick the keeper, plus a fallback chain.

What I'd skip

  • Vosk Telugu small model. WER too high for medical content; produces unusable transcripts on names and dosages.
  • SeamlessM4T as the clinic-deployed model. CC-BY-NC 4.0 forbids commercial use. Evaluation only.
  • Wav2Vec2-XLSR Telugu checkpoints from 2021–2022. Predates the AI4Bharat work; superseded.
  • Running everything on CPU. Possible but a 5-minute visit takes 4–6 minutes to transcribe; you lose Sarvam's main pain point (latency).

Open questions worth answering before committing

  • Where does the GPU live? On-prem at the clinic (Devon-style PC under a desk), or one cloud GPU shared by all clinics? On-prem keeps audio off the network entirely and side-steps DPDP-Act questions but adds a maintenance burden.
  • Are visits ever > 25 minutes? If yes, ASR memory becomes a real concern; pre-chunking with a VAD becomes mandatory (silero-vad is free).
  • How many languages do we actually see? If it's Telugu + English-only, IndicConformer is overkill on languages — can fine-tune a 200M-parameter model and trade some accuracy for 5× lower VRAM.
  • Do we need streaming partial transcripts in the recorder UI? The current product spec deliberately doesn't show live transcript. If that ever changes, IndicConformer's streaming variant or the NVIDIA Citrinet streaming model becomes the right pick.
Practical recommendation. Stand up the local pipeline in parallel with Sarvam — don't replace it yet. Run both on the next 100 real consultations, compare extracted-field accuracy (not just WER) on the Claude output, and only switch when the local stack matches Sarvam on prescription extraction in particular. Doctors will accept slower if the medicine names are right; they'll never accept faster if the medicine names are wrong.
