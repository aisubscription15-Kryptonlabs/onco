1 · What ABDM is, in plain English

ABDM stands for Ayushman Bharat Digital Mission. It was launched by the Government of India in September 2021 and is run by the National Health Authority (NHA). Think of it as India's UPI for healthcare — a national, free, open-API layer that lets every player in the system (clinics, hospitals, labs, pharmacies, insurers, and patients themselves) move clinical data around securely without each building point-to-point integrations.

The mission has four practical goals:

  • One health ID per citizen (ABHA). A 14-digit number that uniquely identifies a patient across hospitals, labs, and insurers. Free, voluntary, based on Aadhaar / mobile / email.
  • One longitudinal record per citizen. Records stay with the facility that generated them; the patient's ABHA app fetches any record they consent to share.
  • One registry of real providers. HPR (doctors) + HFR (facilities) let any system verify credentials.
  • One open standard for the data. FHIR R4 + ICD-10 + LOINC + SNOMED CT + UCUM. No proprietary formats.

3 · FHIR R4 — the wire format ABDM mandates

FHIR (Fast Healthcare Interoperability Resources, pronounced "fire") is HL7's modern healthcare data standard. R4 is the fourth and currently stable major release (Oct 2019). ABDM specifies R4 exactly — not R3, not the in-progress R5.

Where older standards (HL7 v2, CDA) used pipe-delimited or XML message formats, FHIR represents each clinical concept as a resource — a JSON or XML object with a fixed schema, stable URL, and explicit references to other resources. The ~150 resource types cover everything from Patient to ImagingStudy to Coverage.

An OPD consultation typically involves: Patient, Practitioner, Encounter, Condition, Observation, MedicationRequest, AllergyIntolerance, ServiceRequest, and a Composition wrapper that ties them together.

The Bundle

Resources travel together inside a Bundle: 

{
  "resourceType": "Bundle",
  "type": "document",
  "entry": [
    { "fullUrl": "urn:uuid:...", "resource": { "resourceType": "Composition", ... } },
    { "fullUrl": "urn:uuid:...", "resource": { "resourceType": "Patient", ... } },
    ...
  ]
}

For ABDM exchange, Bundle.type = "document" is required — the first entry must be a Composition that anchors all other resources via section.entry[].reference.

5 · What ICD-10 is and why we use it

ICD-10 = International Statistical Classification of Diseases and Related Health Problems, 10th Revision. WHO-maintained, ratified 1990, used worldwide. About 14,000 codes covering every diagnosis, symptom, finding, and external cause of injury.

An ICD-10 code is one letter (chapter) + two digits (category) + optional decimal subdivision:

J 02 . 9
│  │   │
│  │   └─ subdivision (.9 = "unspecified")
│  └────── category (02 = "acute pharyngitis")
└───────── chapter (J = "diseases of the respiratory system")

Sonnet's extraction prompt is instructed to emit 1–3 ICD-10 codes per diagnosis. The doctor sees them on the review screen (with yellow/green markers showing whether the AI inferred them or quoted from the consultation), edits before saving, and they go on the visit row as icd_codes text[].

6 · How MedAssist bundles FHIR as an endpoint API

On-demand. No nightly export, no separate worker — when an authenticated request hits the endpoint, the Bundle is assembled in-memory from existing Postgres rows. 

GET /api/fhir/visit/<visit-id>
Authorization: Supabase session cookie
Response: 200 OK
Content-Type: application/fhir+json; charset=utf-8

Pipeline:

[1] Authenticate → Supabase session cookie verifies user. [2] Authorize → RLS scopes the visit row to the user's clinic. [3] Fetch → 4 parallel queries: visit, patient, doctor, allergies[] [4] Map → lib/fhir/mappings.ts attaches LOINC / UCUM / SNOMED codes, translates Indian frequency shorthand, derives birthDate from age if needed. [5] Assemble → lib/fhir/bundle.ts emits the 9 resource types. [6] Audit → lib/fhir/disclosures.ts logs SHA-256 + bytes to clinical_disclosures (≥7-year retention). [7] Return → JSON.stringify, application/fhir+json.

7 · Consent Manager (HIE-CM)

The Health Information Exchange & Consent Manager is the brokering layer that lets a patient at, say, AIIMS Delhi pull their consultation history from a clinic in Bangalore without either institution exposing its database. It does not store medical records — only signed consent artifacts.

[1] Patient links HIP Patient opens their PHR app, searches "MedAssist", links their record. We receive a discovery callback. [2] HIU requests records Specialist hospital calls CM with the patient's ABHA, scope (e.g. OPConsultRecord), date range, purpose. [3] CM forwards to patient Patient's PHR app prompts: approve / deny. [4] Patient grants CM mints a signed consent artifact (scope, expiry, HIU public key). [5] CM notifies HIP CM POSTs the artifact to our consent webhook; we verify the signature and store the artifact ID. [6] HIU requests data CM tells us "fulfil this consent ID, encrypt to this public key, post to this URL". [7] HIP delivers Bundle We build the Bundle, encrypt with ECDH + AES-GCM, POST to HIU callback. [8] Audit Both ends log the disclosure with the artifact ID, timestamp, scope, content hash.

What "consent" means as data

  • HI types — which of the 8 ABDM record types are shareable.
  • Date range, purpose, frequency, hard expiry.
  • Revocable any time from the patient's PHR app.

Compliance requirements (non-negotiable):

9 · What's done

Everything in this section is committed to the codebase and verifiable by inspection. The data layer that ABDM cares about is complete; the schema is shaped, the extractor populates it, the FHIR exporter emits a conformant Bundle, and audit logging is in place.

Schema

  • Clinics, doctors, patients, visits, appointments, visit_doctors tables with clinic-scoped RLS.Migrations: 20260428115000_initial_setup.sql, 20260428160000_multi_role.sql, 20260503000000_demo_compat.sql
  • Patient ABDM identity — abha_id, abha_address columns + lookup index.Migration: 20260508000000_abdm_fhir_fields.sql
  • Patient FHIR fields — birthdate, given_name, family_name, structured address (address_line1/2, city, state, postal_code, country).Migration: 20260509000000_abdm_full_compliance.sql
  • Doctor HPR ID — doctors.hpr_id column + lookup index.Same migration. UI capture in /(app)/settings.
  • Visit FHIR fields — icd_codes text[], encounter_class (default AMB), field_assumptions jsonb.Migrations: 20260508000000_abdm_fhir_fields.sql
  • Structured allergies — patient_allergies table (allergen, reaction, severity, recorded_at) with RLS.Migration: 20260509000000_abdm_full_compliance.sql
  • Cost tracking — api_usage_events with INR rates for Sarvam, Haiku 4.5, Sonnet 4.6, Opus 4.7.Migrations: 20260507000000_api_usage_events.sql, 20260507000001_cost_inr.sql
  • Disclosure audit log — clinical_disclosures append-only table with content SHA-256, byte count, consumer type, and consent artifact ID column for the post-CM future. Monthly summary view.Migration: 20260510000000_clinical_disclosures.sql

Extraction pipeline (Sonnet / Haiku prompt)

  • Doctor speaker identification from diarized transcript — decisive selection with doctor_id_confidence.
  • All clinical fields extracted: vitals, chief complaints, HPI, exam findings, provisional and confirmed diagnoses, investigations, prescription with diff statuses, advice, follow-up.
  • ICD-10 codes — 1–3 per visit, WHO ICD-10 system. Common Indian OPD codes embedded in prompt for grounding.
  • verificationStatus mapping — provisional vs confirmed routed by spoken language.
  • Investigation phrasing — comma-separated lab-requisition phrasing so each test becomes a ServiceRequest.
  • Medication phrasing — canonical dose units, frequency shorthand, lowercase route names that map cleanly to FHIR.
  • Per-field assumption tagging — every inference-allowed field is marked "stated" or "assumed".
  • Hard rule: no inference on advice, prescription medicines, vitals, follow-up days. Server-side normalizer downgrades any "assumed" on these fields to null as a defense in depth.
  • Prescription diff — new / continued / modified / stopped against previous Rx.
  • Sonnet fallback on low extraction confidence.

FHIR Bundle export

  • GET /api/fhir/visit/[id] route returning application/fhir+json.
  • OPConsultRecord profile claimed on the Composition; DocumentBundle on the Bundle; per-resource profile URLs for Patient, Practitioner, Encounter, Condition, MedicationRequest, Observation, AllergyIntolerance, ServiceRequest.
  • Composition with 9 LOINC-coded sections: Chief complaints, Medical history, Physical examination, Vital signs, Diagnosis, Allergies, Medications, Investigation advice, Advice, Follow-up.
  • Patient resource with ABHA identifier, structured name (given/family), gender mapping (M/F/O → male/female/other), birthDate always emitted (real value or derived from age), structured Address with country defaulting to IN, telecom.
  • Practitioner resource with HPR ID as primary identifier and council registration as secondary.
  • Encounter resource with class AMB, status mapping from visits.status, period, participant.
  • Condition resources with ICD-10 codings, verificationStatus, clinicalStatus.
  • MedicationRequest resources with structured dosageInstruction.timing from BD/TDS shorthand, route mapped to SNOMED CT, doseAndRate.doseQuantity, dispenseRequest.expectedSupplyDuration in UCUM Duration.
  • Observation resources for every vital with LOINC codes (8480-6, 8462-4, 8867-4, 8310-5, 59408-5, 29463-7) and UCUM units.
  • AllergyIntolerance resources from the structured table; falls back to parsing the legacy free-text column.
  • ServiceRequest resources — one per investigation, split from the comma-separated text.
  • Cross-resource references via urn:uuid: Bundle.entry.fullUrl.

UI for ABDM data

  • Patient intake form captures DOB, ABHA ID, ABHA address, given/family-name split (auto from full name with manual override at save), structured address.
  • Doctor settings form captures HPR ID alongside council registration number.
  • Yellow/green field markers on every inference-allowed clinical field (chief complaints, HPI, exam findings, provisional/confirmed diagnosis, ICD codes, investigations, follow-up notes). Fades to neutral once the doctor edits the value.
  • ICD-10 chips — codes render as monospace pills below the input for clarity.
  • Assumption legend banner shown above the form when any markers are present.
  • Provider mentions stripped from user-facing UI ("Sarvam", "Claude", "en-te" all removed; replaced with neutral wording).

Audit & logging

  • Per-call cost log — api_usage_events populated by every transcribe / extract / pre-visit-summary call with INR cost, tokens, audio duration, model used.
  • Disclosure audit — recordDisclosure() in lib/fhir/disclosures.ts writes a row for every FHIR Bundle export with SHA-256 of the released body, byte count, consumer type, actor.
  • Audit failures never break the request — try/catch wraps every audit insert and logs to console only.

Validator harness

  • Synthetic fixture generator — scripts/fhir/generate-fixture.ts creates a complete OPD visit (ABHA-bearing patient, HPR-bearing doctor, full prescription, vitals, allergy, investigation) and emits a Bundle.
  • Validator wrapper — scripts/fhir/validate.sh generates the fixture by default. Set FHIR_STRICT_VALIDATE=1 to download the HL7 Java validator and run against the NRCES IG (ndhm.in#6.5.0 by default) and the OPConsultRecord profile.
  • npm scripts — npm run fhir:fixture, npm run fhir:validate, FHIR_STRICT_VALIDATE=1 npm run fhir:validate:strict.
  • CI workflow — .github/workflows/fhir-validate.yml triggers on changes to lib/fhir/**, types/db.ts, or scripts. Caches the validator jar and IG packages between runs.

10 · What's partially done

  • FHIR Bundle export works for a single visit. Needed for full ABDM: A version of the same export wired to ABDM Consent Manager flow — verifies a CM-issued consent artifact, encrypts the Bundle to the HIU's public key (ECDH + AES-GCM), and POSTs to the requester's callback URL instead of returning JSON to the browser.
  • Disclosure audit log records manual exports today. Needs: Linking each row to a verified consent artifact ID once the consent webhook exists; recording HIU public-key fingerprint and request ID in metadata.
  • Patient address is structured in the schema but the legacy free-text patients.address column still exists and may be used by older rows. The exporter falls back to it. A one-time backfill could split legacy free-text addresses into structured fields; not strictly required for compliance.
  • Allergies are read from the structured table when present; exporter falls back to parsing free-text known_allergies when the table is empty for a patient. No UI yet to add/edit per-allergy rows individually; extraction during a recording does not populate this table.
  • ICD-10 codes are emitted by the AI and editable as a comma-separated field with chip preview. Needed for richer UX: A code-search picker (autocomplete against an ICD-10 lookup) so doctors can swap a .9 "unspecified" code for a more specific match without typing.
  • Verification status is decided by which field (provisional vs confirmed) the AI fills. There is no explicit UI control to flip a diagnosis from provisional to confirmed without retyping it in the other field.
  • Investigation extraction writes a comma-separated string. The exporter splits to one ServiceRequest per item but the items are free text — no LOINC codes attached. ABDM accepts free-text but LOINC mapping is preferred for reporting.

11 · What's pending

These items are not yet started. They are grouped by whether they require external dependencies (NHA sandbox credentials, legal entity) or can be built immediately.

Can be built today (no NHA dependency)

  • ICD-10 code-picker with autocomplete against a bundled ICD-10 dictionary (~14k codes; one-time JSON download in the browser).
  • Per-allergy editor on the patient detail page — add / edit / remove rows in patient_allergies with allergen, reaction, severity.
  • FHIR validator integration tests covering edge cases beyond the happy-path fixture (missing DOB, no ABHA, no HPR, multiple ICD codes, stopped medicines, no allergies, no investigations).
  • Bundle download button on the review screen — currently the FHIR endpoint exists but no UI button calls it.
  • Disclosure audit viewer for admin — read-only table view with filter by patient, date, consumer type.
  • Verification-status flip control in the review screen — single button to mark a provisional diagnosis as confirmed.
  • Encounter.reason mapping — emit Encounter.reasonCode from chief complaints when the doctor presents a single primary reason.
  • SNOMED-CT-IN drug coding (optional but preferred) — map common Indian drug brand names to India National Drug Code via a static lookup table.
  • Server-side address structuring — backfill legacy free-text addresses into the structured columns where pattern-detectable.

Blocked on NHA HIP registration

These all require sandbox credentials and a registered legal entity before any code can be written or tested.

  • ABHA verification flow (M1) — OTP, Aadhaar, mobile-based verification screens. A patient can enter their ABHA at intake and we verify it with NHA before linking.
  • HIP discovery webhook — endpoint that receives "do you have records for ABHA X?" queries from a patient's PHR app. Returns yes/no without leaking patient existence.
  • HIP linking webhook — endpoint that receives the patient's authorisation to attach our clinic to their ABHA record book.
  • Consent notification webhook — POST /api/abdm/consent/notify that receives signed consent artifacts from the CM, verifies the CM's X.509 signature, stores the artifact ID, scope, expiry.
  • Consent revocation webhook — receives revocation notice from CM; marks artifact invalid; denies any subsequent requests against it.
  • Encrypted Bundle delivery — variant of the FHIR export that derives an ECDH shared secret with the HIU's published public key, encrypts the Bundle with AES-256-GCM, and POSTs to the requester's callback URL.
  • X.509 request signing — every outbound API call to NHA signed with our DSC's private key.
  • CM-signature verification — verify NHA's signature on every inbound consent artifact against NHA's published public key chain.
  • Consent dashboard for patients at the clinic — read-only view of active consents, expiry dates, revocation status (issuance happens in the patient's PHR app, not ours).
  • Disclosure audit linkage — every CM-driven disclosure references the consent artifact ID; admin viewer filters by it.

Non-code work

  • Incorporate the legal entity (private limited or LLP). Obtain CIN and GSTIN.
  • Register the clinic on HFR (Health Facility Registry). Online via NHA portal.
  • Register doctors on HPR (Healthcare Professional Registry). Online via NHA portal.
  • Provision a production FQDN with valid TLS (Let's Encrypt or paid). No preview URLs.
  • Buy a Class-3 Digital Signature Certificate from an Indian licensed CA (eMudhra, SafeScrypt, etc).
  • Apply for NHA sandbox access; receive sandbox client credentials.
  • Pass the M1 sandbox test script for ABHA identity flows.
  • Pass the M2 sandbox test script for record linking.
  • Pass the M3 sandbox test script for consent-driven information sharing.
  • Schedule the NHA live demo against the certified profiles.
  • Engage a CERT-In empanelled auditor; remediate findings.
  • Appoint a Data Protection Officer (DPO) and publish contact details.
  • Draft and publish a privacy policy compliant with DPDP Act 2023 (consent receipts, principal rights, breach-notification protocol).
  • Publish a grievance redressal contact integrated with NHA's grievance portal; commit to 7-day response SLA.
  • Submit M3 certification application to NHA. On approval, list on ABDM's public HIP directory and switch from sandbox to live environment.
