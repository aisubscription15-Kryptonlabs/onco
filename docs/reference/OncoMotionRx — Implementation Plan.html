Implementation Plan

Build the production-ready installable application based on the final design screens. Cross-platform mobile for patients, web for clinicians, and web for admins — with a robust, scalable backend.

Version 1.0 June 2026 Target: iOS · Android · Web

1. Executive Summary

OncoMotionRx is a prescribable exercise-therapy platform for oncology patients. This plan defines how to build the production application from the approved high-fidelity designs into a working, installable product.

What We Are Building

  • Patient Mobile App (iOS & Android): Installable native-feel application for daily use, onboarding, prescriptions, sessions, progress tracking, and Artie chat.
  • Doctor Web Application: Browser-based dashboard for oncologists and care teams to prescribe, monitor, review summaries, and receive alerts.
  • Admin Web Application: Browser-based panel for program administrators to manage users, content, configuration, and operational oversight.
  • Shared Backend: Robust API, database, real-time notifications, AI orchestration, wearable integrations, and storage — HIPAA-aligned from day one.

Key Product Decisions

Decision Cross-platform mobile

Use React Native with Expo so a single codebase ships to both iOS and Android while retaining native HealthKit / Google Fit integrations.

Decision Web for doctors & admins

Use React + TypeScript + Vite for both clinician and admin web apps, sharing a component library and API client.

Decision Supabase-first backend

Start with Supabase (PostgreSQL, Auth, Realtime, Edge Functions, Storage) for speed. Provide an AWS migration path for 10,000+ users.

Decision AI companion Artie

Powered by Anthropic Claude (Sonnet for complex reasoning, Haiku for fast responses) with a prompt registry and escalation logic.

Why Mobile-Only for Patients?

  • Wearable integrations (Apple HealthKit, Google Fit) are native-first on mobile.
  • Patients need quick friction-free daily interactions; a pocket app beats a browser bookmark.
  • Push notifications drive adherence for daily prescriptions and session reminders.
  • Voice journaling and live guided walks work best on a device patients carry.

Why Web for Doctors & Admins?

  • Clinical workflows require larger screens, multi-patient views, charts, and printable summaries.
  • Admins manage configuration and review analytics — desktop web is the standard tool.
  • No need to install or maintain separate clinician mobile apps.

Why React Native + Expo?

iOS & Android from one codebase

Reduces mobile development effort by ~60% versus native Swift + Kotlin. One team, one language (TypeScript), consistent UX.

Native integrations

Expo provides managed workflows for push notifications, camera, audio recording, health kit, and background tasks.

OTA updates

Push bug fixes and content updates instantly without waiting for App Store review cycles.

Future-proof

Can eject to bare workflow if advanced native modules are needed later; no rewrite required.

Data Flow Summary

  1. Patient opens mobile app → authenticates via Supabase Auth (email/pass, magic link, or OAuth).
  2. Onboarding collects treatment context, barriers, preferences, and wearable permissions.
  3. Backend generates personalized FITT prescription with MET-hour target.
  4. Daily: app shows prescription card; patient logs activity or syncs wearable.
  5. Activity data flows to backend; adherence engine calculates minutes, MET-hours, adherence %.
  6. Artie sessions run through AI orchestrator; responses are personalized, safety-checked, and stored.
  7. Doctor web app surfaces patient roster, alerts, visit summaries, and adherence trends.
  8. Admin web app manages users, curriculum, prompts, analytics, and exports.

8. Infrastructure Options

Option A: Supabase-First (Recommended for Launch)

Best for: Speed to market, built-in security, teams without dedicated DevOps.

  • Database: Managed PostgreSQL with RLS policies
  • Auth: Supabase Auth with JWT, MFA, OAuth
  • Realtime: WebSockets for alerts and dashboard updates
  • Edge Functions: Deno serverless for AI orchestration, wearable ingestion, PDF generation
  • Storage: S3-compatible for voice recordings and PDFs
  • Limit: Cost-effective to ~10,000 active patients; beyond that, dedicated infrastructure becomes cheaper

Option B: AWS Production-Scale

Best for: 10,000+ users, full compliance control, long-term cost optimization.

  • Database: Amazon RDS PostgreSQL (Multi-AZ)
  • API: ECS/Fargate or EKS running Node.js/Express or Fastify
  • Auth: Cognito or Auth0
  • Storage: S3 with CloudFront signed URLs
  • Realtime: API Gateway WebSockets or ElastiCache (Redis)
  • AI: Direct Anthropic API integration via AWS Lambda or ECS tasks
  • Monitoring: CloudWatch, X-Ray, GuardDuty

Option C: Hybrid (Recommended for Scale)

Best for: Start fast on Supabase, migrate heavy tables to AWS as you grow.

  • Keep Supabase Auth, Realtime, and Storage for speed
  • Move large time-series tables (activity_logs, symptom_reports, ai_interactions) to AWS RDS or TimescaleDB
  • Run compute-heavy AI orchestration on AWS ECS/Lambda
  • Use read replicas for doctor/admin dashboards

10. Scaling Strategy

Phase 1: Launch (0–1,000 users) — Supabase-First

  • Single Supabase project handles auth, database, realtime, storage, and edge functions.
  • Expo mobile app distributed via TestFlight and Google Play Internal Testing.
  • Doctor and admin web apps deployed on Vercel.
  • Focus: product-market fit, onboarding conversion, adherence.

Phase 2: Growth (1,000–10,000 users) — Optimize & Partition

  • Add read replicas for doctor/admin dashboards.
  • Move time-series activity logs to a separate partition or TimescaleDB hypertable.
  • Implement AI response caching for common questions.
  • Add CDN for static assets and voice playback.
  • Introduce feature flags and staged rollouts via LaunchDarkly or PostHog.

Phase 3: Scale (10,000–30,000+ users) — AWS Hybrid

  • Migrate core database to AWS RDS with Multi-AZ and read replicas.
  • Move AI orchestration and PDF generation to ECS/Fargate for predictable scaling.
  • Implement data retention policies: purge voice audio after 90 days, keep transcripts.
  • Shard by site/health system for multi-tenancy.
  • 24/7 monitoring, on-call rotation, SOC 2 / HIPAA audit preparation.

Excess AI usage / runaway API costsHighHard per-user daily/monthly call and token caps; rate limiting per patient; automated alerts when usage spikes; flag accounts exceeding thresholds for immediate review; fallback to cached responses and static content
Wearable sync unreliableMediumManual fallback logging for patients without wearables, clear sync status, retry logic, Apple Health + Google Fit first
App Store rejectionMediumFollow Apple/Google health app guidelines, provide medical disclaimers, avoid unverified health claims
Doctor adoption lowHighMinimal workflow disruption, visit-ready summary, EHR integration roadmap, prescribe in under 60 seconds
HIPAA compliance gapsHighSign BAAs, encrypt everything, audit logs, RLS, regular security reviews, legal review before launch
Scale beyond Supabase limitsLowDesign schema for migration, use UUIDs, avoid Supabase-specific lock-in, plan AWS hybrid at 10K users
