1. Quickstart — getting the app running

  1. Apply the database setup. Open Supabase Dashboard → SQL Editor → paste the contents of supabase/full-setup.sql (or full-setup.txt) and run. Watch the Messages tab — it prints temp passwords for the seeded team.
  2. Disable email confirmation in Supabase. Authentication → Providers → Email → toggle Confirm email OFF. Without this, signups need email verification that won't work in dev.
  3. Start the dev server. From the project root run npm run dev. The app comes up at http://localhost:3000.
  4. Sign in. Use any of the seeded accounts (table below) or create a fresh admin via "Create an account" → "Create new clinic".

2. User accounts you can sign in with

The credentials are at the top of this page. A quick reference of what each role does:

  • Admin — manages the team and the clinic; can also act as a doctor.
  • Doctor — runs consultations, records audio, reviews and prints prescriptions.
  • Medical Assistant — front-desk intake; creates EMRs, captures vitals, routes to doctors.
Rotating passwords. Anyone can change their password via Supabase's "forgot password" flow. Or, as the admin, you can reset passwords directly via Supabase Dashboard → Authentication → Users → click the user → Reset password.

5. Day-in-the-life: Medical Assistant

Medical Assistant

Sai Dharam Tej · saidharamtej@kryptonlabs.ai

  1. Sign in. Use email + temp password SaiDharam2026!.
  2. Land on the front desk. Heading reads "Front desk". You see "Today's intake queue" — every visit created today, regardless of which doctor it's assigned to.
  3. New patient walks in. Click + New EMR.
  4. Step 1 — Patient. Either:
    • Type 2+ characters in Search existing patient — picks up returning patients by name/phone/EMR.
    • Or fill the new-patient form below if it's a first-timer.
  5. Step 2 — Vitals. Optional but recommended. BP / pulse / temp / SpO₂ / weight. Add chief complaint as a free-text reason for visit.
  6. Step 3 — Assign doctors. Check the box next to each doctor who should see this patient today. Pick a role from the dropdown — attending (default), resident, consultant. You can pick multiple if a resident sees them before the consultant.
  7. Save & route. Click Save & route to doctor. The visit lands on the doctors' queues immediately. The pre-visit AI brief starts generating in the background.
  8. Need to update vitals later? Find the visit in your queue, click into it — you land on a vitals + assignment edit screen.
Tip. The faster you fill vitals, the better the AI brief will be when the doctor opens the visit. The brief explicitly highlights abnormal vitals (BP ≥140/90, T ≥100°F, SpO₂ <94%, etc.).

6. Day-in-the-life: Doctor

Doctor

Dr. Venkat / Abhijit / Chiranjeevi

Path A — Patient was prepped by reception

  1. Sign in. Email + temp password (e.g. Venkat2026!).
  2. Open your queue. Heading reads "Patients". Top section is "My queue today" — only visits assigned to you. Each card shows the patient name, vitals already captured (BP, pulse, etc. as pills), and any chief complaint reception noted.
  3. Tap a queue card. If no audio yet, you go to the recording page. The reception note appears as a blue card above the recorder so you remember what the patient came in for.
  4. Hit the big mic button. 25-min cap, pause/resume, live waveform. Stop when done.
  5. Wait ~3–5 min. Three stages run automatically:
    • 🎙️ Audio uploaded
    • 🌐 Sarvam transcribes & translates
    • 🤖 Claude extracts fields
  6. Review screen opens. At the top: the AI pre-visit brief (auto-generated by Haiku 4.5) summarising context. Below: editable fields with AI badges and Revert buttons. Right side: prescription editor with diff against last visit (NEW / CONTINUED / MODIFIED / STOPPED).
  7. If Claude got the doctor speaker wrong: click Flip doctor ↔ patient at the top — re-runs extraction with swapped attribution.
  8. Edit anything that's wrong. Tweak doses, add notes, drop medicines. Editing a continued med's dose auto-flips it to MODIFIED.
  9. Save & print prescription opens an A5 print page in a new tab and goes through the OS print dialog. Print to paper or save as PDF.

Path B — Walk-in, no reception involvement

  1. Click + New EMR from the home page.
  2. Quick patient capture. Search for existing or fill name + age + sex (phone/email optional).
  3. Pick what to do:
    • Save & start recording — auto-creates the visit, assigns you as attending, takes you straight to the recorder.
    • Open blank visit — opens the review screen empty for full manual entry.
  4. Continue as in Path A from step 5.

8. Detailed feature walkthroughs

Recording a consultation

  1. From your queue or "+ New EMR", land on /emr/[id]/visits/new?mode=record.
  2. Big green mic button. Click it — browser asks for microphone permission. Allow.
  3. Live waveform shows audio level. Timer counts up. Pause/Resume work as expected.
  4. Click the red square to stop. The flow auto-progresses through three stages:
    1. Uploading — audio.webm goes to Supabase Storage at {user_id}/{visit_id}.webm.
    2. Transcribing — server downloads the audio, posts to Sarvam batch API, polls every 5 s. Default timeout: 20 min.
    3. Extracting — Claude Sonnet 4.6 with the speaker-labelled transcript + previous prescription as context.
  5. You auto-redirect to the review screen.
If transcription times out: the visit row stays in the database with the audio attached. You can return to it from your queue and the system will retry — it picks up the existing audio file rather than re-recording.

The review screen anatomy

Top: Pre-visit AI brief

Four sections — Patient context, Today, Recent pattern, Watchpoints. Generated once on first open by Claude Haiku 4.5 from vitals + last 2 visits. Refresh button re-generates if vitals or notes were updated.

Speaker-ID banner (only if 2+ speakers detected)

Shows which speaker Claude identified as the doctor and confidence (high / medium / low). One-click Flip button reattributes if wrong.

Left column: Vitals + Clinical fields

Numeric inputs for vitals. Text fields for chief complaints, history, examination findings, diagnosis, investigations. Each field has an "AI" badge if Claude populated it; "Revert" appears if the doctor edited.

Right column: Prescription editor

Each medicine on its own card, color-coded by status:

  • NEW (green) — first-time prescribed.
  • CONTINUED (white) — was on the previous visit's Rx, doctor said continue or didn't mention it.
  • MODIFIED (yellow) — was on previous Rx, dose/freq/duration changed. Shows previous values inline.
  • STOPPED (red, strikethrough) — doctor explicitly stopped, or removed by clicking the X (preserves audit trail).

Drag-handle reorder (up/down arrows). "+ Add medicine" for manual entries.

Collapsible: View transcript

Speaker-labelled English transcript. The doctor's lines are highlighted in green. Useful for verifying what Claude based its extraction on.

Bottom: Doctor's notes

Free text, internal-only. NOT printed by default (configurable via PDF_INCLUDE_DOCTOR_NOTES_BY_DEFAULT).

Sticky save bar

Cancel · Save without printing · Save & print prescription. Saving updates the visit row; opening print launches a new tab with the A5 print page.

Printing a prescription

The print page is a static HTML page styled for A5 portrait:

  • Letterhead (if uploaded) at top, OR a clean default header with doctor name + qualification + reg number + clinic info.
  • Patient block: name, age/sex, EMR number, date.
  • Vitals line.
  • Complaints, examination, diagnosis, investigations as labelled rows.
  • ℞ section with numbered medicines (only those NOT marked stopped).
  • Advice + follow-up date.
  • Signature image (if uploaded) + doctor name + reg number at bottom.
  • "AI-generated draft, reviewed and approved by Dr. {name}." disclaimer line — required for medico-legal.

Press Ctrl+P (or Cmd+P) — set paper size to A5 portrait in the print dialog. Or save as PDF.

Settings

  • Profile fields: name, qualification, registration number, clinic phone.
  • Clinic name & address — also used on the print header.
  • Signature image: upload a transparent PNG. Used on every prescription.
  • Letterhead: if you have a clinic letterhead, upload it. Replaces the default header on prints.
  • Team management (admin only): button at the top, leads to /settings/team.

10. Troubleshooting

SymptomWhat's happeningFix
Login fails with "Email not confirmed" Supabase email confirmation is on; SMTP is unconfigured Toggle Authentication → Providers → Email → Confirm email OFF
"No admin clinic found" when running seed SQL You haven't signed up your admin account in the app yet Sign up via the app first (creates the clinic + admin row), then re-run the seed SQL
Doctor can't see assigned visits visit_doctors row missing If created via doctor's "+ New EMR", auto-assign happens. If from MA's intake, check the assignment list. Worst case, edit the visit's intake page and re-pick the doctor.
Sarvam returns "Diarization not supported in real-time API" You're hitting the sync endpoint instead of batch The current code uses the batch API. Confirm SARVAM_ENABLE_DIARIZATION=true in .env.
Recording flow times out Sarvam queue is busy or audio is > 25 min Bump SARVAM_JOB_TIMEOUT_MS in .env to 1800000 (30 min). The route already declares maxDuration = 800 — fine on local dev, needs Vercel Pro for production.
Pre-visit AI brief shows nothing Visit just created, generation hasn't completed yet Wait 5–10 s and refresh, OR click the "Refresh" link on the brief card.
Print page font looks wrong Google Fonts didn't load (offline?) Use the system default — print still works; Devanagari/Tamil rendering may downgrade.
"Function timeout" on transcribe in production Vercel Hobby caps API at 60s Upgrade to Vercel Pro (800s cap) or refactor to a webhook-driven flow. See CAVEATS.md §1.

Where to look when something breaks

  • Browser console: client-side errors, especially auth and recording.
  • Server logs (terminal running npm run dev): API route errors, Sarvam/Claude failures.
  • Supabase Dashboard → Logs → API/PostgREST: RLS rejections, permission errors.
  • Supabase Dashboard → Storage → visit-audio: confirms audio uploads landed.
  • Supabase Dashboard → Table Editor → visits: see llm_extraction_raw column for full Claude response, useful for debugging extraction quality.
