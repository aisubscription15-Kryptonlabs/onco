MedAssist — STT pipeline: cost × resources × output quality

Comparison of every realistic option for getting Telugu/Hindi-mixed OPD audio into structured EMR fields. Spans the current Sarvam pipeline, three local-Whisper variants, two Gemini paths, GPT-4o audio, and the hybrid options. All numbers are practical April 2026 estimates — measure before committing.

Volume assumption: 3-doctor clinic, 50 visits/doctor/day × 25 days × 12 months = 45,000 visits/year. Single-doctor figures = ÷ 3.

TL;DR — three options worth piloting, in order

  1. Gemini 2.5 Flash audio → JSON (single API call for transcript + extraction). Cheapest, fastest to ship, no GPU. Test first.
  2. Local IndicConformer + Claude Sonnet 4.6 (transcript local, extraction cloud). Best accuracy on Indian medical content. Requires a GPU box. Pays back in ~2 months.
  3. Keep Sarvam as a fallback behind an env flag for the first 3 months on either of the above.

Skip GPT-4o audio (10× more expensive than Gemini for similar quality) and SeamlessM4T (non-commercial license).

7. Detailed write-ups

Option A — Sarvam + Claude (current)

What it is
Audio → Sarvam Saaras v3 batch (translate + diarise) → Claude Sonnet 4.6 (extract) → EMR JSON.
Strengths
Working today. Best diarisation for noisy 3-speaker scenarios. Indian-hosted ASR (DPDP-friendly). No hardware to manage.
Weaknesses
Slowest of all options — batch queue + processing takes 2–5 min. Most expensive sustainably.
When to keep it
For the first 3–6 months, run as a fallback behind STT_PROVIDER=sarvam while you validate alternatives. Don't rip it out.

Option B — Local IndicConformer + Claude

What it is
Self-hosted FastAPI service running AI4Bharat's IndicConformer 600M (Telugu ASR) + pyannote.audio 3.1 (diarisation). Optionally IndicTrans2 for Telugu→English MT, OR send Telugu directly to Claude Sonnet 4.6 (which translates and extracts in one call).
Strengths
Highest Telugu accuracy among open-source models (~10–15% WER). Code-mix native. Fast (30 s end-to-end on RTX 3060). Audio never leaves clinic. Pays back in ~2 months at 3-doctor volume.
Weaknesses
Requires a Linux GPU box (₹40k+ one-time) + ops burden of keeping it alive. Diarisation is a separate model to deploy. ~2-day proof-of-concept effort minimum.
When to pick it
If clinic data sensitivity matters (DPDP-Act, hospital partnerships) or if monthly volume is 5,000+ visits where the GPU pays back fast.

Option C — Gemini 2.5 Flash one-shot ★

What it is
Single Gemini API call: audio in, structured EMR JSON out. Replace both Sarvam and the Claude extract call with one Google call. Add transcript field to the schema so audit trail is preserved.
Strengths
Cheapest end-to-end (~₹0.20/visit). Fast (~60s). No hardware. One vendor. Long context — handles 25-min recordings without chunking. Good code-mix performance.
Weaknesses
Telugu medical accuracy unverified — needs a 10-recording side-by-side evaluation before commit. Diarisation is heuristic, not a dedicated model. Vendor lock-in to Google. Audio leaves the clinic.
When to pick it
If cost reduction is the primary driver and the side-by-side evaluation comes back acceptable on prescription accuracy. The 1-day evaluation is the cheapest experiment available.

Option D — Gemini 2.5 Pro one-shot

What it is
Same as Option C but on the larger Pro model — better reasoning, better Telugu, ~5× the cost of Flash but still 3× cheaper than Sarvam+Claude.
Strengths
Highest cloud-only quality. Stronger structured-output adherence. Deep reasoning helpful for nuanced extraction (e.g. "patient said allergic to penicillin once 5 years ago — note but don't escalate").
Weaknesses
Slower than Flash. More expensive than Flash. Same data residency caveats.
When to pick it
If Option C's evaluation shows Flash dropping drug names or misattributing speakers and you want to stay cloud-only.

Option E — GPT-4o audio one-shot

What it is
OpenAI's audio-native model. Sends audio directly via input_audio content type, returns text/JSON.
Strengths
Strong general intelligence. Good code-mix. Works.
Weaknesses
~10× the cost of Gemini Flash for similar quality. Slower than other options. No India data residency without going through Azure OpenAI.
When to pick it
Skip. Cost makes it a non-starter at OPD volumes. Re-evaluate when OpenAI ships cheaper audio pricing.

Option F — Whisper Large-v3 (translate mode) + Claude

What it is
OpenAI's open-source Whisper running locally with task=translate — outputs English directly. Fed to Claude for extraction.
Strengths
Single ASR step (skips MT). MIT-licensed. Apple Silicon friendly via whisper.cpp. Simplest local setup.
Weaknesses
Telugu WER is ~22–28% (compared to IndicConformer's ~10–15%). Drug names get mangled more often.
When to pick it
Day-1 local proof of concept before committing to IndicConformer. Or as a degraded fallback if IndicConformer container dies.

8. Decision flow

Do you have a Linux GPU box?
  ├── No → Are you willing to pay ~₹9k/year for cloud STT?
  │         ├── Yes → Run Option C (Gemini Flash) eval, pilot if it passes
  │         └── No  → Stay on Sarvam, revisit when GPU is available
  │
  └── Yes → Is your bottleneck cost or accuracy?
            ├── Cost     → Run Option C (Gemini Flash) eval
            └── Accuracy → Run Option B (IndicConformer) eval
                          ├── Drug-name accuracy ≥ Sarvam → Ship Option B
                          └── Drug-name accuracy < Sarvam → Try Option D (Gemini Pro) or stay on Sarvam

9. Recommended action plan

Week 1 — Cheap experiment

  1. 1Take 10 real recordings (different doctors, different languages, different lengths).
  2. 2Run them through current Sarvam + Claude pipeline. Save the extracted JSON.
  3. 3Run the same audio through Gemini 2.5 Flash with a prompt that emits {transcript, extraction}.
  4. 4Manually score: drug names matched? Diagnosis matched? Vitals matched? Speaker attribution correct?

Cost: ~₹100 in API calls + 1 day of work. Decisive answer on whether Option C is viable.

Week 2 — If C passes, ship it

  1. 1Add STT_PROVIDER=sarvam|gemini env flag.
  2. 2Implement lib/ai/gemini-audio.ts that returns the existing SarvamResult shape so nothing else changes.
  3. 3Default to Gemini, fall back to Sarvam on error.
  4. 4Monitor extraction quality for 4 weeks against Sarvam shadow runs.

Week 2 (alt) — If C fails, try B

  1. 1Rent an RTX 4090 on Runpod for 1 week (~₹4k).
  2. 2Containerise IndicConformer + IndicTrans2 + pyannote behind FastAPI.
  3. 3Run the same 10 recordings; compare against Sarvam.
  4. 4If B passes, buy a permanent RTX 3060 12 GB and deploy.

Don't do (yet)

  • Don't tear out Sarvam — keep as fallback.
  • Don't switch to GPT-4o audio at current pricing.
  • Don't deploy SeamlessM4T (CC-BY-NC, non-commercial).
  • Don't deploy Vosk (~40% WER on Telugu).
  • Don't run Whisper on CPU (latency win disappears).

10. Open questions worth answering before any commit

  • Is data residency a hard requirement? If yes, IndicConformer + Claude (Option B) is the only path that keeps audio in India. Gemini and GPT-4o both leave India by default.
  • Are visits regularly > 15 minutes? If yes, Gemini Pro / Flash both handle long context fine. Whisper local needs VAD-based chunking.
  • How often do you have 3+ speakers (family member translating)? If frequent, dedicated diarisation (pyannote / Sarvam) beats Gemini's heuristic.
  • What does the Sarvam contract look like? If you have a fixed monthly fee, the cost case for switching weakens. If pay-per-use, the case is strongest.
  • Does the team have Python ops capacity? Gemini path needs none. IndicConformer path needs someone to monitor a Python service.
Bottom line. The cheapest experiment with the biggest information gain is the Week 1 evaluation against Gemini 2.5 Flash. ₹100 and one day of your time will tell you whether you can save ~₹150,000/year while shipping faster transcripts. Do that first; everything else follows.
