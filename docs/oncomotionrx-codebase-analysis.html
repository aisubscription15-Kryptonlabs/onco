OncoMotionRx Codebase Analysis

Current structure

  • app/(patient) contains patient mobile web routes.
  • components/onco/demo contains frontend-only demo screens and dashboards.
  • lib/onco/demo contains localStorage-backed demo state and actions.
  • components/onco/operations contains the shared admin/app-provider Operations Console shell and AI usage components.
  • app/api still contains legacy backend routes and should be handled in the backend cleanup phase, not this frontend pass.

Landing/login changes

  • app/page.tsx is now a polished OncoMotionRx landing page with embedded DemoRoleSelectPanel.
  • app/demo-role-select/page.tsx reuses the same panel as a standalone role-selection page.
  • /patient/login keeps a patient entry path but directs new patients toward onboarding.

Onboarding privacy logic

  • Self-start captures email/name and sets onboardingStartMode = self.
  • Clinic code captures email and links site without PHI.
  • Patient invite code requires verification before prefill.
  • Store actions document the intended backend contract: submit code, verify invite, confirm profile, and clear invite flow.

Operations console

  • Admin and app-provider screens share layout components while keeping role guards and separate routes.
  • AI usage dashboards provide the next billing/ops reporting surface.

Supabase tables next

  • Core: users/profiles, sites, roles, patients, invite codes, onboarding answers, prescriptions, activities, symptoms, notifications.
  • Operations: AI usage events, report exports, billing rollups, audit events, prompt versions, MET catalog, program sessions.

Readiness notes

  • Web: add PWA manifest, offline state, push-notification opt-in, and production analytics.
  • iOS/Android: prepare Expo adapters for timers, health data, push notifications, voice, clipboard, and sharing.
  • UX: make data tables more mobile-native and add backend error/retry states.
