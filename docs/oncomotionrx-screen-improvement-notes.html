OncoMotionRx demo notes

Screen Improvement Notes

The demo now uses a polished OncoMotionRx landing page, reusable role selector, privacy-aware patient onboarding entry, shared operations shell, and AI usage/reporting dashboards.

Current application structure

  • Patient mobile routes live at /onboarding, /today, /prescription, /sessions, /progress, /artie, /guided-walk, /doctor-summary, /devices, /notifications, and /privacy-sharing.
  • Doctor routes remain under /doctor with roster, alerts, patient details, prescription review, and summaries.
  • Site admin routes remain under /admin and now use the shared Operations Console shell.
  • App-provider routes remain under /app-provider and now use the same Operations Console structure with platform-scoped navigation.

Landing and login changes made

  • / is a two-column OncoMotionRx landing page with the hero message and embedded role selector.
  • /demo-role-select uses the reusable DemoRoleSelectPanel component.
  • Patient login routes users toward /onboarding unless they explicitly continue the existing demo patient.
  • Visible MedAssist login copy has been removed from the active demo screens.

Patient onboarding flow explanation

  • Start on my own captures email and optional name, marks the patient as self-started, and continues from treatment context.
  • Care Code SAM-GVOC-7429 shows masked invitation details, requires demo verification value 1974, then prefills Sam Rivera clinic details.
  • After verified invite confirmation, onboarding resumes at safety/preferences rather than repeating clinic-known treatment context.

Care Code privacy logic

  • The Care Code reveals only masked details before verification.
  • Only Care Code plus verification may prefill name, diagnosis, treatment status, treatments, and assigned doctor.
  • Backend phase should audit every code submission and verification attempt.

Admin/app-provider consolidation plan

  • OperationsShell and OperationsNav provide shared layout while preserving role-specific permissions.
  • Site Admin navigation is site-scoped: overview, team, patients, program config, MET catalog, symptom rules, prompts, analytics, AI usage, billing, and audit.
  • App Provider navigation is platform-scoped: overview, site requests, sites, platform analytics, global AI usage, prompt registry, audit logs, and billing operations.

AI usage/reporting dashboard plan

  • /admin/ai-usage and /app-provider/ai-usage show AI calls, token estimates, cost estimates, cost per patient/site, and cost by agent.
  • Usage table includes filters for date range, site, agent, model, patient, and status.
  • Demo actions include view summary, export usage report, and mark reviewed.
  • Backend phase should derive these rows from immutable usage events and report-generation jobs.

Supabase backend tables needed next

  • profiles, sites, site_members, patient_profiles, patient_site_links, invite_codes, invite_verifications, onboarding_answers, prescriptions, activity_logs, symptom_reports, notifications, ai_usage_events, report_exports, audit_events, and billing_usage_rollups.
  • Role-level policies should separate patient, doctor, care-team, site admin, and app-provider permissions.

iOS/Android/web readiness notes

  • Before React Native/Expo, isolate domain logic and replace browser-only APIs with adapters.
  • Plan HealthKit, Health Connect/Google Fit, Garmin, and Fitbit provider differences before implementation.

Remaining UI/UX improvements

  • Improve responsive data tables with mobile-friendly row cards.
