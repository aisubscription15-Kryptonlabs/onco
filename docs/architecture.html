MedAssist — Architecture

Voice-driven EMR for Indian OPD clinics. The doctor records the consultation, the system transcribes and drafts a structured EMR, the doctor reviews and saves. Everything is ABDM / FHIR R4 ready.

1 · System overview

MedAssist turns a consultation conversation into a clinic-ready EMR in minutes. A medical assistant captures vitals at intake; the doctor records the consult; the audio is transcribed, structured by an LLM, and presented to the doctor for review. The reviewed visit is saved as a Postgres row that maps cleanly to a FHIR R4 OPConsultRecord Bundle for ABDM exchange.

Standards

  • FHIR R4 (HL7)

5 · API endpoints

EndpointPurpose
POST/api/transcribeSend a visit's stored audio to Sarvam Saaras v3. Stores transcript text + diarized speaker turns + detected language on the visit row. Logs cost row.
POST/api/extractRun Claude on the diarized transcript. Identifies the doctor speaker, extracts every clinical field, ICD-10 codes, prescription medicines with diff status, follow-up, and the per-field assumption map. Falls back to Sonnet on low confidence. Persists to the visit. Logs cost row.
POST/api/pre-visit-summaryGenerate a 4-section briefer (Patient context, Today, Recent pattern, Watchpoints) from intake + last 2 visits. Cached on the visit; force: true regenerates.
GET/api/fhir/visit/[id]Emit the visit as a FHIR R4 OPConsultRecord Bundle. application/fhir+json. Logs a row to clinical_disclosures.
POST/api/clinicUpdate clinic profile (name, address, city, state, established year, letterhead). Admin-only.
POST/api/clinic/regenerate-inviteMint a new 8-char invite code. Admin-only; enforced by regenerate_clinic_invite_code() SQL function.
POST/api/team/membersInvite a colleague by email; sends sign-up link with the clinic's invite code prefilled.
PATCH/api/team/members/[id]Change a member's role (doctor / medical_assistant / admin). Admin-only.
DEL/api/team/members/[id]Remove a member from the clinic.
POST/api/emr-numberGenerate a clinic-unique EMR number (e.g. HD-2026-00042). Used at patient creation.

7 · Features by role

Medical Assistant

  • Pending intake queue (visits in intake)
  • New patient + new visit modal with vitals + chief complaint
  • Assign one or more doctors per visit (attending / resident / consultant)
  • Optional appointment booking at intake time
  • Capture/edit ABHA ID, ABHA address, structured address
  • Front-desk vitals re-entry on /intake screen

Doctor

  • Personal queue: queued, in-progress, awaiting-review counters
  • Quick consultation launcher (search patient → start recording)
  • Audio recorder with live waveform (max 25 min)
  • Auto-pipeline: upload → transcribe (Sarvam) → extract (Claude)
  • Review screen with yellow (assumed) / green (captured) field markers
  • Prescription editor with diff against last visit (new / continued / modified / stopped)
  • Speaker flip (re-attribute doctor ↔ patient + re-extract)
  • Pre-visit AI summary (Patient context · Today · Recent pattern · Watchpoints)
  • Print-ready prescription with letterhead + signature
  • FHIR Bundle export (single click) for referral / patient handoff

Admin

  • Clinic profile (name, address, established year, letterhead)
  • Team management — invite by email, change roles, remove
  • Invite-code rotation
  • Cost & usage view (per service, per month, INR)
  • Disclosure audit log (read-only, append-only)

Cross-cutting

  • Dark / light theme toggle (no flash, persisted in localStorage)
  • Keyboard-friendly modals + focus management
  • Mobile drawer navigation
  • Toast notifications for success / error / info
  • Search-as-you-type patient lookup
  • Audio retention 30 days; transcript + structured fields kept indefinitely

8 · Key flows

8.1 New visit (MA-led)

MA → /(app)/dashboard → "New patient" modal ↓ POST /api/emr-number (mint clinic-unique HD-2026-NNNNN) INSERT patients (ABHA ID, address, age/DOB, sex, allergies, conditions) INSERT visits (status='queued', vitals, chief_complaints) INSERT visit_doctors[] (assigned doctors with role) INSERT appointments[] (optional, per assigned doctor) void POST /api/pre-visit-summary (background — Claude Haiku, cached on visit row)

8.2 Doctor records consultation

Doctor opens patient → "Record consultation" ↓ Browser: MediaRecorder captures webm/opus Upload to Storage bucket `visit-audio` (path: ${doctorId}/${visitId}.${ext}) UPDATE visits SET audio_url = path, status='in_progress' ↓ POST /api/transcribe (Sarvam Saaras v3 batch — translate=en, diarize=true) UPDATE visits SET transcript_text, transcript_speakers, transcript_language INSERT api_usage_events (sarvam_stt, audio_duration_seconds, cost_inr) ↓ POST /api/extract (Claude Haiku 4.5 default; Sonnet 4.6 fallback if low conf.) Identifies doctor speaker, extracts every clinical field, emits ICD-10 codes + per-field assumption map. UPDATE visits SET , status='awaiting_review' INSERT api_usage_events (claude_*, input/output tokens, cost_inr) ↓ Redirect → /(app)/emr/[id]/visits/[vid]/review

8.3 Review and save

Review screen renders: • PreVisitSummary (4-section briefer) • SpeakerBanner ("Identified Dr. X as SPEAKER_1 — flip if wrong") • AssumptionLegend (green = captured, yellow = inferred) • EditableFields with `assumption` prop coloring each field • PrescriptionEditor with status-coloured rows (new/continued/modified/stopped) • TranscriptPanel (collapsible) Doctor edits as needed → "Save" ↓ UPDATE visits SET , status='completed', completed_at=now() ↓ Optional → "Save & print" → /print → window.print()

8.4 FHIR export (manual today; consent-driven post-ABDM)

GET /api/fhir/visit/[id] ↓ 4 parallel queries: visits, patients, doctors, patient_allergies buildOpConsultBundle() in lib/fhir/bundle.ts → Composition with 9 LOINC-coded sections → Patient (ABHA id, structured address, derived birthDate) → Practitioner (HPR id + council reg) → Encounter (class=AMB, period, participant) → Conditions[] (ICD-10 + verificationStatus) → MedicationRequests[] (Timing, Duration, SNOMED route, doseQuantity) → Observations[] (LOINC + UCUM for every vital) → AllergyIntolerances[] (clinicalStatus + criticality) → ServiceRequests[] (one per investigation) ↓ recordDisclosure({ consumer_type: 'fhir_export', sha256, bytes }) ↓ Response: 200 application/fhir+json

9 · AI pipeline (STT → extraction → review)

Sarvam Saaras v3 (transcription)

  • Batch API — entire audio uploaded once, response is full transcript + diarized turns.
  • Translate mode: input audio in any Indian language, output English text.
  • Diarization tags (SPEAKER_1, SPEAKER_2…) attached per turn.
  • Cost: ₹45 per hour of audio (≈ ₹0.0125 / sec).

Claude extraction prompt (lib/claude/prompts.ts)

Six-step instruction:

  1. Identify the doctor speaker. Decisive — must commit to one speaker if any clinical content is present. Confidence in doctor_id_confidence.
  2. Attribute correctly. Patient symptoms vs. doctor findings/diagnosis.
  3. Extract. Strict JSON; no preamble; never invent; never hedge inside clinical fields.
  4. Tag inference. Per-field field_assumptions map: stated / assumed / null. Inference is forbidden for advice, prescription, follow_up_days, vitals.
  5. ICD-10 code. 1–3 WHO ICD-10 codes for the most likely diagnosis. Common Indian OPD codes listed in the prompt for grounding. verificationStatus = confirmed vs. provisional decided by doctor's spoken language.
  6. Prescription diff. Status flag per medicine against previous Rx: new / continued / modified / stopped.

Yellow / green markers

Every inference-allowed field returns a marker. The EditableField component reads it and styles the input accordingly: green = AI quoted the conversation, yellow = AI inferred from context (please verify). Once the doctor edits a field so its value differs from the AI's suggestion, the marker fades to neutral. Prescription medicines, advice, and vitals are never tinted yellow — the system disallows AI inference for those.

Pre-visit summary

A separate Claude call (Haiku 4.5) produces a ≤180-word brief in 4 sections — Patient context, Today, Recent pattern, Watchpoints. Cached on the visit row; regenerated only on intake update.

10 · FHIR / ABDM layer

Three modules in lib/fhir/:

  • mappings.ts — static lookup tables: SYSTEM URLs, sex → FHIR gender, BD/TDS shorthand → Timing.repeat, vital → LOINC + UCUM, route → SNOMED CT, frequency → repeat, duration → UCUM Duration, visits.status → Encounter.status. Includes deriveBirthDate() that falls back to YYYY-01-01 from age so Patient.birthDate is always present.
  • bundle.ts — buildOpConsultBundle() assembles the FHIR R4 Bundle. ~400 LoC; emits Composition with 9 LOINC-coded sections + cross-referenced resources via urn:uuid: entries.
  • disclosures.ts — append-only audit logger. SHA-256 of every released Bundle is logged, never the body.

The validator harness lives in scripts/fhir/:

  • generate-fixture.ts — synthesize a realistic Bundle from canonical inputs.
  • validate.sh — download HL7 validator_cli.jar on first run, validate against NRCES IG.
  • .github/workflows/fhir-validate.yml — CI on every push touching lib/fhir/**.

11 · Auth & RLS model

Supabase email + password auth. The auth.users.id matches doctors.id (1:1 — every authenticated user has a row in doctors, regardless of clinical role).

RLS policies (clinic-scoped)

  • current_clinic_id() — security-definer function returns the caller's doctors.clinic_id.
  • Every clinic-owned table has a single SELECT/INSERT/UPDATE policy: clinic_id = current_clinic_id().
  • set_clinic_id_from_user() trigger backfills clinic_id on insert so app code never has to.
  • Admin-only operations (regenerate invite, change roles, update clinic) are gated by SQL functions with auth.uid() role checks.

12 · UI system & theming

Tailwind CSS v3 with darkMode: "class". Brand palette from Mantine blue, accent from Mantine teal, dark scale (ink) warmer than Tailwind's default slate so dark-mode surfaces don't feel cold.

  • Reusable classes — .card, .btn-primary/secondary/ghost, .input-base/flush, .badge-*, .nav-link[-active], all with paired .dark variants.
  • Theme toggle — ThemeScript component in <head> applies the dark class before paint. Persisted in localStorage (hd-theme).
  • App shell — desktop sidebar (260 px) + mobile hamburger drawer.
  • Audio waveform — canvas-rendered, reads document.documentElement.classList.contains("dark") at draw time so it adapts to theme switches without a re-render.

13 · Cost tracking (INR)

Every Sarvam and Claude call writes a row to api_usage_events. Costs computed at insert with these constants in lib/usage.ts:

ServiceRate (INR)
Sarvam Saaras v3₹45 / hour audio (≈ ₹0.0125 / sec)
Claude Haiku 4.5 — input₹85 / 1M tokens
Claude Haiku 4.5 — output₹425 / 1M tokens
Claude Sonnet 4.6 — input₹255 / 1M tokens
Claude Sonnet 4.6 — output₹1,275 / 1M tokens
Claude Opus 4.7 — input / output₹1,275 / ₹6,375 per 1M (not used in default flow)

14 · Audit & compliance logs

  • Cost log — api_usage_events for every external API call.
  • Write-side audit — audit_events for visit create/update/delete.
  • Disclosure log — clinical_disclosures for every clinical-record release (append-only, ≥7-year retention via absent UPDATE/DELETE policies).
  • RLS — every table has clinic-scoped policies; service role used only for privileged inserts (cost log, disclosure log).
