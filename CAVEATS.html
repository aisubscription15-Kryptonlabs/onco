Hello Doctor — Operational Caveats

Things that work in dev but need a decision before / right after going to production. Each item names the file or env var that controls it.

Status: the project is currently not deployed anywhere — running on local npm run dev. The "Vercel" caveats below only matter if and when you decide to host the app on Vercel. They are kept here so they don't get lost when you eventually deploy.

1. Vercel function timeouts (deployment-only)

Doesn't apply right now — local next dev has no timeout cap, so any recording length works. Only relevant if you deploy to Vercel.

Long-running API routes (/api/transcribe, /api/extract, /api/pre-visit-summary) declare export const maxDuration = … in seconds. Vercel enforces a hard cap per plan:

PlanDefault capMax requestable
Hobby (free)60 s60 s
Pro90 s800 s
Enterprise90 s900 s

"Hobby" is Vercel's free tier — the one you get on signup with no payment method. If you ever push this to vercel.com on the free plan, the transcribe endpoint will be killed after 60 seconds, which means anything longer than ~3 minutes of audio will fail (transcription typically takes 10–30% of audio duration plus queue time). Pro is $20/month and lifts that to 800 s.

  • /api/transcribe is set to maxDuration = 800. Will fail on Hobby for any recording over a few minutes. Move to Pro before testing long recordings on Vercel, or refactor to a webhook flow (Sarvam can call back on completion).
  • /api/extract is set to maxDuration = 300. Fine on Pro.
  • /api/pre-visit-summary is maxDuration = 60. Fits Hobby.

If you stay on Hobby: raising SARVAM_JOB_TIMEOUT_MS won't help — Vercel will kill the function first. The fix is the webhook approach.

If you host somewhere else (Render, Fly, Railway, your own box): no fixed function timeout, this whole caveat is moot.

2. Sarvam batch API timing

The batch API is a queue-and-process pipeline, not synchronous:

  • Queue wait before processing starts: usually under 60 s, can spike to a few minutes during peak load. We can't see queue position, only poll status.
  • Processing time after queueing: roughly 10–30% of audio duration. So a 15-min recording typically processes in 2–5 minutes; 25 min in 3–8 minutes.
  • Polling cadence: 5 s (SARVAM_POLL_INTERVAL_MS). This is fine — Sarvam's quota is generous on status checks.
  • Max audio length: capped at 25 min in the recorder UI (TRANSCRIPTION_MAX_MINUTES). Sarvam's batch API officially supports much longer files but this matches our timeout budget.

3. Audio format

Browser MediaRecorder writes audio/webm;codecs=opus by default (Safari falls back to audio/mp4/AAC). Sarvam's batch API accepts both formats per their current docs, but if you start seeing 400 unsupported audio errors:

  1. Check the actual mime type in Supabase Storage on the visit-audio bucket.
  2. If WebM is rejected, the cleanest fix is to convert to WAV server-side before upload — needs fluent-ffmpeg plus an ffmpeg static binary on the server. Vercel's filesystem is read-only, so on Vercel you'd need to package the binary or use a service like Replicate.

4. Auth provider config

We use email + password today, not phone OTP. To make this work cleanly:

  • In Supabase dashboard → Authentication → Providers → Email: turn "Confirm email" OFF for development. Otherwise, signup succeeds but the user can't log in until they click an email link, and Supabase's dev SMTP delivery is flaky.
  • For production: turn confirm-email back on, and configure SMTP under Authentication → Email Templates. Customize the templates to match the clinic brand.
  • To switch to phone OTP later: configure Twilio (or MessageBird) under Authentication → Providers → Phone, then change the login flow in app/login/page.tsx to call signInWithOtp({ phone }).

5. PDF generation

The print page (/emr/[id]/visits/[vid]/print) uses a print-optimised HTML page + browser window.print() — the user prints to PDF or paper from the OS print dialog. Why not server-rendered PDF?

  • We can fully support Devanagari / Tamil / Telugu / Bengali via Google Fonts loaded on the page (already wired up).
  • Server-rendered PDFs (@react-pdf/renderer) require bundling .ttf files for every Indian script. Each font adds ~250 KB to the deployment.
  • If you later need API-level PDF generation (e.g. emailing the prescription as an attachment), use Puppeteer-Core + Chromium-AWS-Lambda and reuse the print page's HTML.

6. Database type generic

The Supabase clients (lib/supabase/{browser,server,admin}.ts) are created without the Database generic. Reason: postgrest-js v2.x does not match a hand-written Database interface unless every named row type satisfies Record<string, unknown> exactly, which TypeScript only allows for type aliases (not interfaces). Everything works at runtime; we cast data as Patient / as Visit after select('*') and use update(... as never) on the few admin-write paths.

To opt into strict typing later: run supabase gen types typescript --project-id <ref> and import the generated Database from there. The generated types satisfy the postgrest constraint.

7. Audio retention

AUDIO_RETENTION_DAYS=30 in .env is descriptive, not enforced. Nothing deletes audio yet. Two options:

  • Cron job in Supabase (recommended): under Database → Cron, add a daily job that runs select … from storage.objects where created_at < now() - interval '30 days' and deletes.
  • Cleanup on visit save: simpler but unreliable.

Until you wire one of these up, audio accumulates indefinitely in Supabase Storage. This applies regardless of where the app is hosted — Supabase is your storage backend in dev too.

8. RLS policies and the admin client

lib/supabase/admin.ts uses the service role key and bypasses RLS — only used inside server-side API routes (/api/transcribe, /api/extract, /api/pre-visit-summary). Never expose service role calls to the client. Audit: if you grep "supabaseAdmin" and find a usage in a "use client" file, that's a leak.

9. Pre-visit AI summary cost

/api/pre-visit-summary calls Claude Haiku 4.5 with prompt-caching on the system prompt. After the first call, subsequent calls hit cache (75% cheaper input). The visit row caches the result in pre_visit_summary so we don't bill twice unless the doctor clicks Refresh.

Order-of-magnitude per visit: <500 input tokens (cached) + 200 output tokens = roughly 0.05 ¢ per visit on Haiku 4.5.

10. Multi-clinic users

A user account belongs to exactly one clinic today (doctors.clinic_id). If you need a single doctor to work at two clinics, the cleanest path is two accounts (different emails). A real solution would split doctors into users + clinic_memberships (many-to-many) — moderate refactor.

11. Migration order

Migrations in supabase/migrations/ are timestamped and need to be applied in order. After cloning a fresh project:

  1. Run 20260428115000_initial_setup.sql first.
  2. Then 20260428160000_multi_role.sql.
  3. Then 20260428170000_rename_role_to_medical_assistant.sql.
  4. Then 20260428180000_seed_team.sql (optional seed).
  5. Then 20260503000000_demo_compat.sql (adds visits.status, visits.completed_at, appointments table, and richer clinic fields used by the new dashboard / appointments UI).

The Supabase CLI (supabase db push) does this automatically; the dashboard SQL Editor does not — paste them in order. Or paste supabase/full-setup.sql (or full-setup.txt) once; it now includes all five migrations and is idempotent.

A .txt copy of every .sql file lives next to it in supabase/ for easy paste into the Supabase Dashboard SQL Editor.

12. Visit status workflow

visits.status is the source of truth for dashboard counters. It is set automatically by:

  • Intake screens — app/(app)/emr/new/ReceptionIntakeForm.tsx writes queued; the new patient modal writes intake if no doctor is picked, queued otherwise.
  • Recording start — DoctorQuickIntakeForm.tsx writes in_progress when the doctor picks "Save & start recording", queued for "Open blank visit".
  • AI extraction — app/api/extract/route.ts flips the row to awaiting_review when Claude returns (won't downgrade an already-completed visit).
  • Review save — app/(app)/emr/[id]/visits/[vid]/review/ReviewScreen.tsx writes completed plus completed_at = now() when the doctor hits Save.

If you add a new code path that creates a visit, set status explicitly so the queue counters stay correct.

13. Appointments

Appointments live in their own table with clinic-scoped RLS (same pattern as patients / visits). The list and booking modal at /appointments use the browser Supabase client directly (no API route needed for CRUD). Day view is anchored to the ?date=YYYY-MM-DD query param; absent it, today.

The doctor dashboard pulls today's appointments filtered to the signed-in doctor; the admin dashboard surfaces them in the recent-activity timeline.

14. Invite-code regeneration

POST /api/clinic/regenerate-invite calls the regenerate_clinic_invite_code function (security-definer, admin-only). The current code stops working immediately on regeneration — there is no grace period. The admin dashboard and clinic-settings panel both expose the regenerate button.
